##  1 文件

### 1.1 文件命名

1、软件可以用长文件名，但长度不超过16字节（不包括扩展名）；

2、文件名和目录名只能使用小写字母、数字和下划线；

3、文件名格式是“模块名_功能名"的组织格式，例如（tcp_send.c)；

### 1.2 文件组织结构

1、 文件组织是以模块为单位的，一个模块一个独立的源代码和资源文件目录，一般结构如下：

2、 一个模块一般包含4种文件：

* 工程文件（.mk）；
* 头文件（.h）；
* 源文件（.c）；
* RC文件（.rc）；

3、一个模块一般只有一个工程文件（）；

4、一个模块的头文件一般分为4部分；

5、c文件一般按照功能模块进行划分，一个模块还可以拆分成多个子模块，

### 1.3 文件格式

* 任何一个文件都应该有头部说明

```c

```



### 1.4 文件大小

为增加代码的可读性和可维护，一个.c文件不宜过大。建议一个源文件不要超过5000行（NBNC规范，No Bug No Crash，是嵌入式/工业级软件开发领域一套高可靠性编码规范）

### 1.5 文件引用

1. 禁止采用extern外部声明方式引用模块外部的任何函数和变量；
2. 引用任何外部函数、数据结构、变量和宏都应该首先引用对方提供的公共头文件（xxx_pub.h）；
3. 引用外部公共头文件可以直接使用文件名，不必包括其相对路径，其路径已经在工程文件Makefile中指定；
4. 引用内部头文件需要包括其相对路径，以本模块的根目录为根；
5. 头文件常常彼此有引用关系，由各个模块自己保证其相对依赖的引用关系；
6. 一般的情况下，**禁止在头文件中再引用其他的头文件 **，一个模块自己的xxx_pub.h引用自己的xxx_drv.h除外；
7. C文件应该只引用需要的头文件，禁止包含多余头文件；
8. 禁止引用.c文件

 ### 2.2 复杂数据类型

结构

```c
typedef struct tagPPP_CB
{
    ...
}PPP_CB_S;
```

联合

```c
typedef union tagSNMP_Cmd
{
    ...
}SNMP_CMD_U;
```

枚举

```c
typedef enum tagSNMP_ErrCode
{
    ...
}SNMP_ERR_E;
```

例子

```c
typedef ULONG OBS_HANDLE;    // 重命名

typedef	ULONG (*IF_DIAG_PF) (IN IF_DIAG_PARA_S *pstArg, IN ULONG ulExecID);  // 函数指针
```

注意：

- 所有的复杂数据结构都以"tag"为前缀，”tag"后一般要紧跟模块名，并以下划线分开后续的结构名，形式如同 **"tagXXXX_XXX"**；
- 重定义typedef的类型一般形式为"XXX_XX_X"，必须全部为大写字母，其中：前两部分一般就是模块名加结构名，一般就采用前面锁列"tagXXXX_XXX"中"tag"后面的部分；
- 由下划线分割的额最后一个字母为后缀，一般用"_S"表示结构，"_U"表示联合，"_E"表示枚举、"_PF"表示函数指针，"_HANDLE"表示句柄资源。
- 数据结构中的元素个数要适中，若结构中元素个数过多，可以依据某种原则拆分成若干个子结构；
- 数据结构设计中，应该合理安排元素个数过多，使其易于理解，并尽可能节省空间；
- 联合类型的各字段必须具备相同的本质特性，禁止把毫不相关的字段组成联合类型；
- 避免数据结构之间通过“虚拟字段”直接访问，保证数据结构的封装性和完整性。

```c
typedef struct tagXXX_DisplayRPCReply
{
    ULONG	ulEntryNum;		/* 传回的表项个数 */
    CHAR	szEntryBuf[0];	/* 返回的字符串 */
}XXX_DISPLAYRPCREPLY;
上述结构通过szEntryBuf引用该结构后继内容，破坏了数据结构的封装性。禁止使用。
```

- 同一模块禁止定义大量雷同的数据结构，要求进行复用和优化。

| ------------------------------------------ | ----------------------------------------------------------- |

1. g_、a、p可以用在其他前缀之前，当有多个重复时，**采用"g_ap"顺序定义**。
2. sz前缀仅用于字符数据，不可使用于字符串指针类型；
3. 应当将修饰词 * 和 & 紧靠变量名。
4. 变量名用驼峰命名法，最长不超过32字节，并反映出实际意义，以增强可读性。

### 3.2 类型转换 

不同基本类型的变量之间赋值时，要有显示的类型转换操作；

```c
ULONG 	ulUserInput - 1;
USHORT	usWIn;
usWIn = ulUserInput;		// 非法
usWIn = (USHORT)ulUserInput;		// 正确
```

如果在类型转换时，要同时转换字长和有无符号类型，则需要明确指出转换的顺序；

例如在不同的操作系统和编译器下，如下的代码的结果是不确定的

```c
ULONG ulTemp;
SHORT sTemp = -2;

ulTemp = (ULONG)sTemp;		// 同时转换字长和符号类型可能会有不同的结果
ulTemp = (ULONG)((USHORT)sTemp);	/* sign cast , then size case ,result 0xFFFF */
ulTemp = (ULONG)((LONG)sTemp);	/* size cast , then sign result 0xFFFFFFFF*/
```



### 3.3 移位 

如果能够预见一个变量在今后会做移位操作，除非是有特殊的目的，否则不要将将这个变量定义成有符号数。因为有符号数对于移位操作来说，很可能导致结果不可预测。

例如：

``` c
char cTwoLength;
cTwoLength >> 4;
/* 当cTowLength为39时，其结果是一个正数，而当其为153时，结果是一个负数*/
```

### 3.4 变量 

1. 所有变量**使用前必须初始化**，严禁将没有初始化的变量作为右值或者入参。（不必在最开始就一定初始化）
2. 变量初始化建议采用“就近原则”，即在开始使用的地方进行初始化，特别是循环控制变量。以减少维护时（如：大函数中修改循环条件）遗漏变量初始化的技术。
3. 每个变量只用于单一用途。
4. 避免具有隐含/多重含义的变量/参数。如下例中 ulNextIndex 为零时表示获取索引错误，否则表示下一个合法索引值。随着取值范围的变化 ulNextIndex 的意义发生了变化。

```c
ulNextIndex = ABC_GetNextIndex(ulCurIndex);
if(0 == ulNextIndex)
{
    /* 获取索引失败 */
}
```



### 3.5 常量 

1. 代码中不能直接使用常量，而应该尽可能使用const生成的常量或宏， 例如：

```c
/* 可读性差 */
if(Trunk[index].trunk_state == 0)
{
    Trunk[index].trunk_state = 1;	/*  */
}
/* 正确做法，除了使用宏以外，也可以使用枚举增加代码的自检能力，减少错误的机会 */
#define TRUNK_IDLE	0
#define TRUNK_BUSY	1
if(Trunk[index].trunk_state == TRUNK_IDLE)
{
    Trunk[index].trunk_state = TRUNK_BUSY;
}
```

2. 可以使用const常量，以保持强类型检查，但是C语言中不能作为维数来定义数组。例如：

```c
const USHORT IDLE_TIMEOUT = 1000;
if(usCurTimes < IDLE_TIMEOUT)
{
    usCurTime++;
}
```

3. 在使用宏来代替常数时，应该包括显示的类型转换，例如，对于BOOL_TRUE和BOOL_FALSE定义。

```c
#define BOOL_T	USHORT
#define BOOL_FALSE	((BOOL_T)0)
#define BOOL_TRUE	((BOOL_T)1)
```



### 3.6 负数 

在编码中要尽量避免使用负数，更不要用“-1”代替“全1”。

```c
ULONG ulTemp = -1;
USHORT usTemp = -1;
/* 以上非法，要使用全1 */
ULONG ulTemp = 0xFFFFFFFF;
USHORT usTemp = 0xFFFF;
```



## 4 宏 

### 4.1 使用宏的场合 

使用宏可以增加代码的可读性，一般在如下场合时可以使用宏：

* 代表一些有明确含义的常数或状态；

* 预编译来实现对各种环境的兼容；

* 代替一些大量被重复使用的代码块；
  * 注意：将简单的小函数改编成宏可以增加执行的效率，但会增加目标代码的规模。而且要注意，由于宏在跟踪调试时非常不方便，如果逻辑比较复杂，还是以函数行为存在为佳。
  * 对于要写成宏的代码块，应该先写成小函数进行调试，在确保没有问题后，再改编成宏。
  
* 可以使用内联inline函数代替宏定义以加强类型检查；
  * 注意：内联函数是通过省去了函数调用的开销，来提高其执行效率的，会因赋值代码而增加目标代码的规模。以下情况不宜使用内联函数：
  
    （1） 函数体内的代码比较长，使用内联将导致内存消耗代价较高。
  
    （2） 函数体内代码的执行时间要比函数调用的开销大。

### 4.2 宏命名 

宏在命名时要以**模块名带下划线开头，全部大写**并以下划线分割，例如：

```c
#define OS_QUE_NUMBER	1000
#define OS_TIMER_STATE_STOP	BOOL_FALSE
```



### 4.3 宏定义 

1. 定义宏时，要将整个宏表达式用括号包含，如：

```c
#define RECTANGLE_AREA(a,b)  ((a)*(b))
```

2. 宏定义包括多条语句时必须使用大括号包含
3. 在宏中如果需要使用局部变量，应该尽量避免和调用者重复，添加特殊的前缀，如：

```c
#define ABC_CALCULATE_SOMETHING(pstA, pstB) \
{											\
	DLL_NODE_S *calcsthNodeTemp;			\
	...										\
    pstB = calcsthNodeTemp;					\
}			
```

4. 用宏简化对复杂数据结构字段的访问，有宏封装的数据结构字段禁止不通过宏而直接操作。如：

```c
#define MBUF_GET_RECEIVE_PORT_IFNET_INDEX(pstMBuf) \
    ((pstMBuf)->stUserTagData.stCommonInfo.ulReceivePortIfIndex)
```

4. 避免重复定义宏。

```c
#define RPR_ITEM_VALID          1      /* rpr_drv.h   L386和L115重复定义 */
#define RPR_ITEM_INVALID        0      /* rpr_drv.h   L384和L114重复定义 */

#define RPR_MACLEARN_HASHTBL_LENGTH  1024  /* rpr_maclrn.h  L33和rpr_def.h  L636重复定义 */
#define RPR_MACITEM_NOTEXIST         3     /* rpr_maclrn.h  L34和rpr_def.h  L637重复定义 */
#define RPR_MACLRN_AGING            1     /* rpr_maclrn.h  L35和rpr_def.h  L638重复定义 */
#define RPR_MACLRN_NOAGING          2     /* rpr_maclrn.h  L36和rpr_def.h  L639重复定义 */
#define RPR_ITEM_VISITED            1     /* rpr_maclrn.h  L38和rpr_def.h  L641重复定义 */
#define RPR_ITEM_UNVISITED          2     /* rpr_maclrn.h  L39和rpr_def.h  L642重复定义 */
#define RPR_MBUF_RESERVEHEAD_LENGTH 0     /* rpr_core.h    L41和rpr_def.h  L46重复定义 */
```

### 4.4 裁减宏 

组件化项目使用编译链接方式进行裁剪，不再使用模块宏的裁剪方式，已有的模块宏会逐步的清除。严禁私自定义任何其他裁剪宏。编译裁剪的情况下，直接在交叉编译文件中使用-D特性来使用，代码中可以直接判断#ifdef

```c
_VERSION_RELEASE
   
// 引用时，注意不能使用如下方式
#ifdef _VERSION_DEBUG
    ...
#else
    ...
#endif
```



## 5 函数 

### 5.1 函数命名 

函数名建议都以“大写模块名_"开头，以大小写交错的形式，表达出函数的实际意义，一般为动宾词组。例如：

```c
VOID NAT_ProcessInputPacketQueue();
BOOL_T ATM_PVCIsUp (ATM_PVC_S *pstPVC);
ULONG MEM_Free();
```

常用关键字命名举例：（XXX为平台模块名， YYY为具体的功能描述）

1. USR_XXX_YYY，由驱动实现
2. XXX_DRV_YYY，
3. XXX_CLI_YYY，
4. XXX_CMD_YYY，
5. XXX_CFG_YYY，
6. XXX_RPC_YYY，
7. XXX_IPC_YYY，IPC相关处理函数，例如 MAC_IPC_CallBack.

#### 5.1.1外部函数

基础或者lib库等公共基础函数和外部接口函数命令建议采用：**"类名（特性名或模块名）_" + "函数功能描述" 两级命名模式**，以便于方便阅读和书写。

例如：

```c++
VOID	STQ_Init		(IN STQ_HEAD_S *pstList);
STQ_NODE_S*	STQ_First	(IN STQ_HEAD_S *pstList);
VOID *	FIB_AllocRTMsg	(IN UINT	uiLength);
VOID	FIB_FreeRTMsg	(IN VOID	*pRTMsg);
```

外部接口函数尽量使用**简单数据类型**作为入口参数类型或返回值类型，尽可能避免使用复杂的结构和数据类型定义，这样可以不必把过多的数据结构定义暴露在组件外面。

一个模块或特性的公共基础函数命名也可以采用上述模式。例如：

```c++
VOID	NAT_ProcInputPktQueue (VOID);
BOOL_T	ATM_PVCIsUp		(ATM_PVC_S *pstPVC);
```

#### 5.1.2 特性函数

**专用于某些场合或适配接口的特定函数名建议采用：“特性名或模块名_” + “关键标识” + “_函数功能描述" 三级命名模式，以方便识别函数的使用环境。**

### 5.2 参数设计 

在函数定义时，函数参数说明必须加上IN，INOUT，OUT表示传输方向的显示说明，并尽可能将参数和函数放在同一行，一行放不下时，按如下方式书写：

```c
VOID XXX_SendFunction( IN VOID *pMsg,
                       IN ULONG ulParal,
                       IN UCHAR ucFlag,
                       INOUT ULONG *pulDataLen,
                       OUT ULONG *pulOutput)
{
    ...
}
/* IN OUT INOUT 仅仅是用来增加代码可读性的宏，都被定义成空 */
```

1. 参数设计要体现函数接口的抽象性，并且参数之间应保持正交性。如果有重叠的属性考虑删除一些
2. 函数的各参数按照IN、INOUT、OUT的顺序排列。
3. 若多个函数采用了相同的一组参数，要保证各函数中这组参数的顺序一致。
4. 明确对函数参数的合法性检查是由函数的嗲用着还是由函数本身负责。缺省是有函数调用者负责。为提高函数的健壮性，模块间接口要求函数本身也要进行参数合法性检查。
5. 尽可能保证函数的参数不超过7个，如果超过则说明函数之间的耦合太紧密，建议做解耦操作。
6. 禁止将函数的参数作为工作变量。
7. 任何函数必须有函数原型声明，对于模块内部函数的原型可以存放在模块的内部头文件，对于仅本C文件引用的函数，建议在该C文件头部申明各函数原型。

### 5.3 返回值 

1. 当一个函数要用返回值来表示执行“成功”或“失败”时，这个函数的返回值应该被定义成ULONG，并且，以“0”表示成功，以“非0”值表示失败的原因。
2. 当一个函数用来做判断，并返回“true”或者“false”时，函数返回值应该被定义成BOOL_T，并且以“1”表示“true”，以“0”表示“false”
3. 避免直接在判断条件中直接使用一个值来判断true或者false，而是用if(bool_true == bFlagOn)这种形式。
4. 执行不会失败或者失败后没有影响时公共基础函数的返回值，可以不判断简化代码。
5. 有可能执行失败并且有后继影响时公共基础函数返回值必须处理。
6. 调用有返回值的模块函数时要求必须处理器返回值。如果执行不会失败或者失败后没有影响时要求明确用（VOID）明示。如果一个函数很多地方需要用到（VOID），则该函数的设计可能需要改善。
7. 除特殊需求外，必须使用等于或者不等于操作成功时的返回值来判断。
8. 返回值为操作成功时，输出参数的有效性应该得到保证。
9. 必须使用被调用函数提供的返回值列表中的值进行返回值判断。

### 5.4 函数构建 

1. 抽象函数的本质需求，仔细权衡和设计函数形参。
2. 函数的功能职责明确而单一。
3. 主次流程分离，控制函数规模和复杂度。
4. 努力消除和降低代码片段间耦合。
5. 严格区分函数层次（C文件内部函数、模块内部函数、模块外部函数），明确参数检查策略和方法。
6. 以正常流程融合异常情况处理。
7. 集中控制或在公共路径上处理资源申请释放。
8. 努力消除和避免“虚假”分支。
9. 所有相关操作成功以后再进行后继操作，避免不必要的回退处理。
10. 紧密相关的功能可以封装成为一个函数实现。
11. 理顺代码流程，避免荣誉操作，比如重复调用其他模块接口获取或进行操作等。
12. 除特殊需求外，被调函数返回非成功时，调用函数不应该处理输出参数，特别是资源型参数。
13. 如果多段代码重复完成同一功能，那么在函数的划分上可能存在问题，可考虑把此段代码进行封装，构造成一个新的函数或者宏等。
14. 除特殊情况外，避免在成对的系统资源操作之间产生异常退出分支，特别是资源申请释放、信号量PV操作、中断开关操作灯。对称操作之间应该尽量可能最小化，严禁将与对称操作无关的功能或逻辑在对称操作区间中实现。
15. 非调度函数应减少或避免使用控制参数，尽量只使用数据参数。

### 5.5 显式返回声明

即便是函数返回值定位为VOID，所有函数必须有一个显示的return声明。

### 5.6 函数大小

为了增加代码段的可读性和可维护性，函数不建议过大，一般不超过200行（NBNC）

### 5.7 函数设计

1. 一个组件对应一个或多个.c源文件及同名.h头文件（可选），以及一个用于公开组件外部接口定义的组件.h头文件。

   **组件的外部接口使用extern声明**，该函数内部负责检查入口参数的合法性。

   **组件的内部接口函数声明在与.c同名的.h文件中**，用于给组件内的其他c文件提供访问接口，其入口参数的合法性由调用者保证，无需检查，使用断言即可。

2. 外部接口函数尽量使用简单数据类型作为入口参数类型或返回值类型，尽可能避免使用复杂的结构和数据类型定义，这样可以不必把过多的数据结构定义暴露在组件外面。

   外部接口函数如果必须使用复杂的数据结构定义入口参数或返回值类型，这些定义放在组件的.h头文件中。

## 6 断言

### 6.1 使用场合

* **正常错误流程**：代码运行中实际会进入的错误分支。例如，收到的协议报文中长度字段的值超出MBUF的范围，这种异常情况在恶意攻击或者友商实现出错时就会碰到，这种情况下应该对报文的长度字段的合法性检查进行错误处理。

* **异常错误流程**：代码运行中不应该进入的错误分支。例如，定义了一个枚举类型作为协议状态变量，并以此做数组下标，调用对应的状态机函数；主要我们代码编写正确，就不会对状态变量赋非法值，这种情况下对协议状态变量的合法性检查就应该使用断言。

1. 断言一般用来检查函数入口的参数，特别是模块内部之间的函数接口，连续顺序调用的函数接口；
2. 已经使用正常代码判断保护的地方不必再增加断言；
3. 必须使用OS_DBGASSERT实现断言，禁止在中断处理中使用断言。
4. **禁止将断言作为错误处理，断言仅在DEBUG版本有效。**
5. 禁止对逻辑表达式以外的其他任何表达式使用断言机制，布尔常量例外。
6. 在代码中尽量不要做任何假设（特别是外部假设），如果一定要做，必须用断言检验假设是否成立。

​	假设可能没有任何语法上的约束，因为设计/需求等变化而不成立，但是依赖的地方却浑然不知。

7. 在正常错误流程，不应该用断言，应该用条件判断等进行错误处理，而且一般还需要提示或调试信息以及清理动作等，比如OS_Malloc申请内存失败。
8. 允许在异常错误流程中同事使用断言和调试信息。因为运行过程中可能会关闭调试，导致这些错误信息无法输出，断言有助于调试阶段发现问题，调试信息有利于快速定位问题。
9. 在维护阶段对代码进行任何修改，都要在DEBUG版本运行并进行充分测试，以保证断言的有效性。



## 7 编写风格

### 7.1 换行

1. 一行只写一条语句，不允许吧多个语句写在一行中；

2. 一行代码的长度不应该超过120个字符，超长的代码应该换行；

3. 如果换行，应该按照如下原则操作：

   1. 在逗号","处换行；
   2. 在操作符后换行；
   3. 长表达式要在低优先级操作符处换行，操作符放在行尾；
   4. 换行后，新行的起始位置应该和上一行对应级别处对齐；

   例如：

   ``` 
   ulResult = ((ulParameter1 + ulParameter2 + ulParameter3 + ulParameter4)/
   			(ulParameter5 + ulParameter6 + ulParameter7 + ulParameter8)) *
   			(ulParameter9 - ulParameter10);
   ```

### 7.2 缩进

1. 在函数体的开始、类的定义、结构的定义、枚举的定义以及if、for、do、while、switch、case语句中的程序都要采用缩进，缩进的距离是4字符；
2. 禁止直接使用"Tab"键来缩进对齐，必须使用空格，因为tab键在不同的编辑器上的缩进是不一致的，会降低代码的可读性

### 7.3 大括号

1. if、else、for、do、while、switch、case、default等语句自占一行，其后的执行语句，一律要用{}括起来，哪怕只有一句；
2. 大括号"{"或"}"自己独占一行，并且成对的"{"和"}"的缩进要对齐，并与引用它的语句左对齐。

### 7.4 空格

1. 关键字（语句）后面跟着“("时， 前面建议有一个空格，函数名后不跟（为SI的界面显示）。
2. 在使用逗号“，”分割多个参数时，逗号“，”后面必须有一个空格。
3. 所有的双目操作符，除了“**.**"和"**->**"以外，两边都必须各有一个空格，而自加"++"和自减"--"运算符和他们的操作对象之间不能有空格。
4. "for"语句的声明的表达式之间应该用空格分割；
5. 强制类型转换后面应该有一个空格；
6. 加空格的目的是使代码更加清晰，所以在已经非常清晰的语句中可以不加空格；
7. 足够清晰的情况下，括号内侧左括号后面和右括号前面可以不加空格，多重括号之间可以不加空格；
8. 一般情况下，不要连续加两个以上的空格。



### 7.5 空行

空行用来分割不同的代码段以增强代码的可读性，以下情况应该使用空行：

1. 函数之间；
2. 局部变量定义和第一条可执行语句之间；
3. 相对独立的程序块之间。

### 7.6 注释

注释通常用来对代码进行概括和补充说明。注释只能包括对阅读和理解代码有关的信息。

1. 必须按照C语言的风格，使用"/*"和"\*/"来添加注释，而不能使用"//";
2. 短注释可以跟在代码行的后面，但必须离代码足远；如果多个短注释出现在同一类型的代码后面，他们应该尽可能保持风格一致（如缩进都相同）；
3. 长注释应该加在其所描述的代码段之前，和前面的代码用空行分开，和所描述的代码段保持相同的缩进；
4. 一个文件的注释率（注释行占总行数的百分比）应该在30%以上；
5. 注释要和代码保持一致，**无用的注释要及时删除**，避免使用二义性语言描述；
6. 对于所有有物理含义的变量、常量，如果其命名不是充分自注释的，在声明时都必须加以注释，说明其物理含义，变量、常量、宏的注释应放在其上方相邻位置或右方；
7. 数据结构声明（包括数组、结构、类、枚举等），如果其命名不是充分自注释的，必须加以注释，对数据结构的注释应放在其上方相邻位置，不可放在下面；对结构中的每个域的注释放在此域的右方。
8. 全局变量要有较详细的注释，包括对其功能、取值范围、哪些函数或过程存取它以及存取时注意事项等的说明；
9. 不要在一行代码内部插入注释；
10. **禁止使用\#if 0 和 /\* \*/方式保留废弃的代码或注释，废弃的代码和注释必须删除。**
11. 代码维护记录注释要求：

``` 
只增加/修改一行代码，采用短注释方式时格式如下
/* Added/Modified by name + workerID, time of questionID */

一段代码采用封闭型格式
/* Begin Added/Modified by name + workerID, time of questionID */
/* End Added/Modified by name + workerID, time of questionID */
```



### 7.7 goto

1. 尽量避免使用"goto";
2. 如果无法避免，必须使用"goto"时，goto的符号应该遵从：函数名+"_"+标签名，标签名的首字母应该大写。

例如：

```c
VOID XXX_Init(VOID)
{
    ...
    if (...)
    {
        goto XXX_Init_Failed;
    }
    
    ...
    if(...)
    {
        goto XXX_Init_Failed;
    }
    
    return ERROR_SUCCESS;
    
XXX_Init_Failed:
    return ERROR_FAILED;
}
```



### 7.8 相等判断

如果需要判断一个变量和一个常量是否相等，必须把常量放在"=="的左边，而将变量放在"=="右边。这样可以避免错误的将"=="写成"="，因为"变量=变量"可以编译通过，而"常量=变量"不能。

判断一个常量和一个函数的返回值是否相等，也需要将常量放在左边。

另外，对"!="的处理也是如此

### 7.9 switch

1. case后面应该使用{}将其包含的代码括起来；
2. 当一个case分支没有break语句时，如果这个case包含语句，在应该出现break地方添加一条注释说明此case没有break；如果这个case不包含任何语句，则在同一行后面应该添加注释说明。
3. 每个switch都必须包含default分支。如果default分支不应该被走到，在其中添加断言声明。

## 8 缺陷预防

### 8.1 字符串类

1. 数据源不可靠时字符串拷贝类操作必须使用有长度限制的Strncpy/OS_strncat系列函数。
2. 数据源不可靠时字符串拼装类操作必须使用OS_nsprintf以确保缓冲区不会越界。
3. 避免重复定义字符串，公用字符串应该公共使用。包括代码组装字符串和公用提示信息。字符串格式化逻辑（如%s  %d等）必须和OS_sprintf一起出现，以保证PCLint检查发挥作用，尽量避免**用枚举或资源ID等动态的方式定义字符串格式化逻辑。**可以使用资源文件来保存字符串格式符逻辑，但必须仔细核对参数个数和类型，避免书写错误。
4. 字符串比较、拷贝操作无特殊要求时必须用OS_str系列函数。
5. 无拷贝需求时，禁止随意拷贝字符串。
6. 严禁大量重复低效率的字符串操作。
7. 在没有必要的情况下，避免初始化整个字符串数组，特别是当数组规模较大时。
8. 存储信息，而不是存储数据：传递信息，而不是传递数据。

### 8.2 数据结构类

1. 相同或同构的数据结构操作应抽象出一套操作函数，而不是每种操作一套函数。相同的数据结构操作是指对同一个数据结构中字段的填写、获取、拷贝、字节序转换或连贯操作。同构的数据结构操作是指对异种数据结构的关键字段（组）的同类或相同操作等。
2. 除特殊需求外，数据结构变量初始化必须先采用OS_Mem_Zero等手段整体清零所有字段，数据结构变量各字段的缺省值要求为零。
3. 除特殊需求外，禁止将对数据结构中的一组字段的幅值操作分散。
4. 对复杂数据结构的清零长度和内存长度等复杂计算或关键逻辑值建议采用结果复用的方式。
5. 避免反复以多重间接寻址方式访问数据结构。
6. 禁止以直接偏移的方式拼装数据消息。
7. 除特殊需求外，禁止以批量丹哥字段赋值的方式进行数据结构赋值，要求使用MEM_Copy

### 8.3 内存资源类

1. 除了特殊需求外，申请内存后必须对内存进行清零操作。
2. 全局指针或结构中的指针释放内存后，必须清零，以防止野指针。
3. 除特殊需求外，第一时间释放内存资源。
4. 集中控制和管理内存资源。
5. 在同一个层级上处理内存资源。
6. 万事具备，方才申请内存资源。
7. 除特殊需求外，要求函数的内存规则简化：即任何情况下，函数都释放（或不释放）内存。
8. 合理选择堆栈内存和动态内存资源。

### 8.4 语言编程类

1. 函数中单个局部变量的占用空间不得超过1K、所有局部变量的总和不得超过2K。对于临时使用的不大于256字节的存储空间禁止申请动态内存。
2. 禁止大量或非核心代码直接访问全局变量。全局变量的访问应该严格受控，对其访问应该通过宏或者函数等封装手段。
3. 只在本C文件中使用的函数和变量，建议使用STATIC修饰符，以严格限定其作用域，避免被无用，使该类函数和变量不进行全局链接，从而加快链接速度。
4. 函数的参数如果为只读属性，特别是只读的字符串参数，建议使用const修饰符，以加强语法检查，避免被误用。
5. 完善和优化代码流程，严格控制函数的异常回退处理分支个数，除特殊需求外，每函数不得超过2个。异常回退处理分支是指需要释放资源或恢复数据或摘除链表节点等清理操作的异常分支。
6. 避免使用默认优先级，用括号明确表达式的运算顺序。
7. 不要使用难懂的技巧性很高的语句，例如三目运算符。
8. 严禁实现与库函数相同的功能。
9. 必须显式地明确操作目标，以免书写错误或理解错误。
10. 避免仅仅为打印断言增加虚假异常分支。

### 8.5 中级 PCLint 检查

1. 新增或修改代码必须经过中级PCLint检查，并对每一个告警逐个确认和消除。（若个别告警确实无法消除，需写入PCLint报告。

### 8.6 调试信息类

本处的调试信息是指模块自我诊断用的程序内部信息，非最终用户可见、可用。

1. 调试信息要求简洁明了（减少代码段内存消耗）、关键和重要信息齐全，能够快速定位问题。禁止将函数名和变量名写入调试信息。（调试信息的输出用驼峰命名，不用空格隔离）
2. 输出调试信息的接口必须进行抽象和封装，并且易于使用和避免错误。
3. 每个函数中调试代码的比例不得超过30%，整个模块调试代码的比例不得超过20%。

## 9 编程思路
