# 内存占用过多问题定位

主要问题有

内存专题

内核地址空间的分布

arm32

```
2^32 = 4GB

0x00000000 ~ 0xBFFFFFFF		3GB		用户态虚拟空间（每个进程独立）
0xC0000000 ~ 0xFFFFFFFF		1GB		内核态虚拟空间（全局共享）

内核起始 0xC0000000
├─ 内核代码/段/全局变量/页表        （几十MB）
├─ Direct 线性映射区（LowMem）      （约700~800MB）
│   👉 永久映射低端物理DDR，直接访问
├─ vmalloc 动态内存区              （128MB左右）
│   👉 内核动态分配、模块加载
├─ ioremap 设备物理寄存器映射区     （预留IO窗口）
│   👉 所有外设寄存器 都在这里映射
└─ 临时kmap窗口 / 保护页 / 空洞
```



arm64

只用48位虚拟地址，高低地址严格分割，空间非常充足

```
虚拟地址范围(48位)                 归属
0x0000_0000_0000_0000 ~ 0x0000_FFFF_FFFF_FFFF    用户态
0xFFFF_0000_0000_0000 ~ 0xFFFF_FFFF_FFFF_FFFF    内核态

FFFF_0000_0000_0000
├─ 线性直接映射区（整个全部物理DDR）
│  ✅ 不管8G/16G/32G 全部永久一对一映射
│  ✅ 彻底消灭 HighMem
├─ vmalloc 区域（超大）
├─ 外设 ioremap 预留区（大把地址随便映射外设）
├─ 内核代码、模块、页表、fixmap 固定映射
└─ 各种预留、安全隔离空洞
```



arm32中的高端内存，实际上是内核地址空间无法映射完所有1G DDR所造成的，DDR中无法被映射的部分，我们称之为“高端内存”—— HighMem

高端内存的本质：有物理页，无永久内核虚拟地址

内核在1G的地址空间中，预留了一块地址 —— 临时映射区/kmap窗口

kmap()

* 把一个HighMem物理页，临时映射到内核虚拟地址
* 返回一个合法内核指针，直接读写

kunmap()

* 用完销毁映射，释放临时窗口





# linux下定位方法

 step1：先观察系统内存使用情况，确认真的内存在下降

```
free 命令
-k, --kibi          show output in kibibytes
 -m, --mebi          show output in mebibytes
 -g, --gibi          show output in gibibytes
     --tebi          show output in tebibytes
     --pebi          show output in pebibytes
 -h, --human         show human-readable output

[root@H3C ~]# free -h
              total        used        free      shared  buff/cache   available
Mem:          7.7Gi       1.0Gi       4.7Gi       1.9Gi       2.0Gi       4.7Gi
Swap:            0B          0B          0B

[root@H3C ~]# free -m
              total        used        free      shared  buff/cache   available
Mem:           7891        1070        4807        1922        2012        4827
Swap:             0           0           0

total：总内存容量
used：应用程序实际使用的内存
free：完全空闲的内存（未被任何用途占用）
shared：临时文件系统（tmpfs）使用的内存
buff/cache：缓冲区和缓存（可以被释放）
available：真正可用的内存（最准确）
swap：交换分区的大小
```

根据其现象来确认是什么问题

| 现象 | 可能原因 | 优先级 |
|------|---------|--------|
| Used 持续增长 | 进程内存泄漏 | 🔴 高 |
| Available 持续下降 | 进程内存泄漏 | 🔴 高 |
| Buffers/Cached 持续增长 | 缓存膨胀 | 🟡 中 |
| Swap 使用率高 | 内存压力大 | 🟠 中高 |





查看具体的进程

```
ps aux  // 显示所有包含其他使用者的行程

USER         PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
bin        70516  0.3  7.8 788296 310732 ?       Ssl  Apr18  18:52 /sbin/portsecd --SupportWebauth=1 --SupportVSI=1 --SupportBAGG --SupportMicroSegment --SupportDot1xOfflineDetect=1 --SupportRemoteMacMove --Dot1xMulticastDefault=0 --Dot1xUnicastDefault=1

USER：进程所属用户
PDI：进程唯一ID，进程号
%CPU：进程占用CPU百分比
%MEM：进程占用物理内存百分比
VSZ：虚拟内存大小，单位（KB），进程申请的全部虚拟地址空间，包含：共享库、swap、未使用内存、文件映射等，不代表实际物理内存消耗。
RSS：常驻内存大小，单位（KB），进程真正占用的物理内存（不含交换分区、未使用内存、共享库不计重复）
TTY：关联终端。 ？代表无终端
STAT：进程状态码。
	S：可中断睡眠（等待事件、IO、定时器，大部分后台进程常态）
	s：会话首进程（session leader）
	l：多线程进程（clone 创建过子线程）
	+：前台进程组，绑定当前终端
START：进程启动时间/日期
TIME：进程累计CPU占用时长
COMMAND：启动进程的完整命令+路径

排查内存泄漏优先看RSS
```





## 内存门限

linux

```
# 查看所有zone内存水位
cat /proc/zoneinfo | grep -A5 -E 'min|low|high'

high:内存回收停止线，对应交换机Normal恢复门限
low：开始后台回收（kswapd），对应交换机minor告警门限
min：强制阻塞回收、极度紧张，对应交换机severe/critical临界门限

整机全局内存门限（全局保底阈值）
1.系统强制保留最小空闲内存（全局critical底线）
cat /proc/sys/vm/min_free_kbytes     // 等于critical阈值
echo "$(cat /proc/sys/vm/min_free_kbytes) / 1024" | bc   // 计算成MB直观查看
```

交换机门限

```
Minor：一级门限
Severe：二级门限
Critical：三级门限
```
