# 内存专题

## 内核地址空间的分布

### arm32

```
2^32 = 4GB

0x00000000 ~ 0xBFFFFFFF		3GB		用户态虚拟空间（每个进程独立）
0xC0000000 ~ 0xFFFFFFFF		1GB		内核态虚拟空间（全局共享）

内核起始 0xC0000000
├─ 内核代码/段/全局变量/页表        （几十MB）
├─ Direct 线性映射区（LowMem）      （约700~800MB）
│   👉 永久映射低端物理DDR，直接访问
├─ vmalloc 动态内存区              （128MB左右）
│   👉 内核动态分配、模块加载
├─ ioremap 设备物理寄存器映射区     （预留IO窗口）
│   👉 所有外设寄存器 都在这里映射
└─ 临时kmap窗口 / 保护页 / 空洞
```



### arm64

只用48位虚拟地址，高低地址严格分割，空间非常充足

```
虚拟地址范围(48位)                 归属
0x0000_0000_0000_0000 ~ 0x0000_FFFF_FFFF_FFFF    用户态
0xFFFF_0000_0000_0000 ~ 0xFFFF_FFFF_FFFF_FFFF    内核态

FFFF_0000_0000_0000
├─ 线性直接映射区（整个全部物理DDR）
│  ✅ 不管8G/16G/32G 全部永久一对一映射
│  ✅ 彻底消灭 HighMem
├─ vmalloc 区域（超大）
├─ 外设 ioremap 预留区（大把地址随便映射外设）
├─ 内核代码、模块、页表、fixmap 固定映射
└─ 各种预留、安全隔离空洞
```



### 高端内存

arm32中的高端内存，实际上是内核地址空间无法映射完所有1G DDR所造成的，DDR中无法被映射的部分，我们称之为“高端内存”—— HighMem

高端内存的本质：有物理页，无永久内核虚拟地址

内核在1G的地址空间中，预留了一块地址 —— 临时映射区/kmap窗口

kmap()

* 把一个HighMem物理页，临时映射到内核虚拟地址
* 返回一个合法内核指针，直接读写

kunmap()

* 用完销毁映射，释放临时窗口



## 内存分配器



### 基本伙伴分配器

连续的物理页称为页块（page block），阶（order）是页的数量单位，2的n次方个连续页称为n阶页块，满足如下条件的两个n阶页块称为伙伴（buddy）。

1. 两个页块是相邻的，即[物理地址](https://zhida.zhihu.com/search?content_id=219643633&content_type=Article&match_order=1&q=物理地址&zhida_source=entity)是连续的；
2. 页块的第一页的物理面页号必须是2的n次方的整数倍；
3. 如果合并（n+1）阶页块，第一页的物理页号必须是2的括号(n+1)次方的整数倍。

伙伴分配器分配和释放[物理页](https://zhida.zhihu.com/search?content_id=219643633&content_type=Article&match_order=4&q=物理页&zhida_source=entity)的数量单位也为阶（order）。以单页为说明，0号页和1号页是伙伴，2号页和3号页是伙伴。1号页和2号页不是伙伴？因为1号页和2号页合并组成一阶页块，第一页的物理页号不是2的整数倍。

#### 分配过程

1. 查看是否有空闲的n阶页块，如果有，则直接分配；如果没有，则进入步骤2。
2. 查看是否有空闲的n+1阶页块，如果有，把该页块拆分为两个n阶页块，一个插入空闲n阶页块链表，另一个分配出去；如果没有，执行步骤3。
3. 查看是否有空闲的n+2阶页块，如果有，拆分为两个n+1阶页块，一个插入空闲n+1阶页块链表，另一个执行步骤2；如果没有，继续寻找更高阶页块。

#### 释放过程

释放过程与分配过程基本相反。

1. 查看伙伴页块是否空闲，如果不空闲，则把这个页块插入到空闲n阶页块的链表中；如果空闲，就合并为n+1阶页块。
2. 如果合并，寻找合并后的n+1阶页块是否空闲，如果空闲，按照步骤1执行。

### 分区伙伴分配器

内核在基本伙伴分配器的基础上做了进一步扩展。

- 支持内存节点和区域，称为分区的伙伴分配器
- 为预防内存碎片，把物理页根据可移动性分组
- 针对分配单页做[性能优化](https://zhida.zhihu.com/search?content_id=219643633&content_type=Article&match_order=1&q=性能优化&zhida_source=entity)。为减少处理器之间的竞争，在内存区域增加一个每处理器页集合

内存区域的结构体struct zone成员free_area用来维护空闲页块，数组下标对应页块的阶数。[结构体](https://zhida.zhihu.com/search?content_id=219643633&content_type=Article&match_order=2&q=结构体&zhida_source=entity)free_area的成员free_list是空闲页块的链表，nr_free是空闲页块的数量。内存区域的结构体成员managed_pages是伙伴分配器管理的物理页的数量。

### 区域水线

首选的内存区域在什么情况下从备用区域借用物理页？此问题从区域水线讲解深入理解，每个内存区域有3个水线。

#### 水线分区

1. 高水线（HIGH）：如果内存区域的空闲页数大于高水线，说明该内存区域的内存充足；
2. 低水线（LOW）：如果内存区域的空闲页数小于低水线，说明该内存区域的内存轻微不足；
3. 最低水线（MIN）：如果内存区域空闲页数小于最低水线，说明该内存区域的内存严重不足。

最低水线以下的内存称为紧急保留内存，在内存严重不足的紧急情况下才会给少量的紧急保留内存使用，或是释放更多的内存来使用。这种情况一般很少存在，现在内存一般足够大，基本不用考虑这个问题。

#### 水线计算

计算水线时，有两个重要参数

1.min_free_kbytes最小空闲字节数。默认值4 l o w m e m _ k b y t e s 4\sqrt{lowmem_kbytes}4
lowmem_kbytes，并且限制范围由机器决定。其中lowmem_kbytes是低端内存大小，单位是KB，通过命令cat proc/sys/vm/min_free_kbytes查看，也可以修改。

2. watermark_scale_factor水线[缩放因子](https://zhida.zhihu.com/search?content_id=219643633&content_type=Article&match_order=1&q=缩放因子&zhida_source=entity)。默认值10，通过命令cat proc/sys/vm/watermark_scale_factor查看，也可修改，取值范围[1,1000]。

可以通过`_setup_per_zone_wmarks()`负责计算每个内存区域的最低水线、低水线和高水线。

- min_free_pages=min_free_pages所对应的页数
- lowmem_pages=所有低端内存区域伙伴分配器管理的页数总和
- [高端内存](https://zhida.zhihu.com/search?content_id=219643633&content_type=Article&match_order=1&q=高端内存&zhida_source=entity)区域的最低水线=zone->managed_pages/1024
- 低端内存区域的最低水线=min_free_pages*zone->managed_pages/lowmem_pages

低水线和高水线计算

```text
增量=max(最低水线/4,managed_pages*watermark_scale_factor/10000)
低水线=最低水线+增量
高水线=最低水线+增量*2
```

如果`最低水线/4`比较大，那么计算公式简化为

```text
低水线=最低水线*5/4
高水线=最低水线*3/2
```

### slab分配器

Buddy提供以page为单位的内存分配接口，这对内核来说颗粒度还太大，所以需要一种新的机制，将page拆分为更小的单位来管理。
linux中支持的主要有：slab、slub、slob。其中slob分配器的总代码量比较少，但分配速度不是最高效的，所以不是为大型系统设计，适合内存紧张的[嵌入式系统](https://zhida.zhihu.com/search?content_id=219643633&content_type=Article&match_order=1&q=嵌入式系统&zhida_source=entity)。在配备大量物理内存的[大型计算机](https://zhida.zhihu.com/search?content_id=219643633&content_type=Article&match_order=1&q=大型计算机&zhida_source=entity)上，slab分配器的管理[数据结构](https://zhida.zhihu.com/search?content_id=219643633&content_type=Article&match_order=1&q=数据结构&zhida_source=entity)的内存开销比较大，所以设计了slub分配器。这里只介绍slab，其余两种差不多，可以类推。

```
free 命令
-k, --kibi          show output in kibibytes
 -m, --mebi          show output in mebibytes
 -g, --gibi          show output in gibibytes
     --tebi          show output in tebibytes
     --pebi          show output in pebibytes
 -h, --human         show human-readable output

[root@H3C ~]# free -h
              total        used        free      shared  buff/cache   available
Mem:          7.7Gi       1.0Gi       4.7Gi       1.9Gi       2.0Gi       4.7Gi
Swap:            0B          0B          0B

[root@H3C ~]# free -m
              total        used        free      shared  buff/cache   available
Mem:           7891        1070        4807        1922        2012        4827
Swap:             0           0           0

total：总内存容量
used：应用程序实际使用的内存
free：完全空闲的内存（未被任何用途占用）
shared：临时文件系统（tmpfs）使用的内存
buff/cache：缓冲区和缓存（可以被释放）
available：真正可用的内存（最准确）
swap：交换分区的大小
```

根据其现象来确认是什么问题

| 现象 | 可能原因 | 优先级 |
|------|---------|--------|
| Used 持续增长 | 进程内存泄漏 | 🔴 高 |
| Available 持续下降 | 进程内存泄漏 | 🔴 高 |
| Buffers/Cached 持续增长 | 缓存膨胀 | 🟡 中 |
| Swap 使用率高 | 内存压力大 | 🟠 中高 |
