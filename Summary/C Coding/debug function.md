# 嵌入式DEBUG方法



## 一 通用方案

## 1 历史环境收集

一个问题出现的时候，首先要在现场将能收集到的条件都收集出来，此项主要是为了避免问题是偶现问题，后续复现困难。

收集的信息可以如下：

* 问题组网，怎么连接的，跑了什么协议
* 出问题时的手法：触发此问题前做了什么操作导致的，还是设备自己运行就出
* log信息：设备诊断信息收集，相关模块的error log，相关模块的当前状态等



### 2 问题复现固化

* 核心目的：将问题的场景和现象分析清楚。没有调查就没有发言权，问题不调查清楚就无法清晰定位。

1. 摸清问题：例如复现的环境（温度、功率等）、组网（与其他设备如何连接）、概率（必现/偶现）；
2. 简化场景：删除部分条件观察复现情况，找到最小复现场景，例如只执行某段代码或最小组网就能复现
3. 精确记录：例如，上电40s时挂死，挂死前设备有xx初始化的打印。

### 3 问题分类

根据问题现象将分类，例如：

| 现象特征                | 大概率问题类型                   | 排查优先级 |
| ----------------------- | -------------------------------- | ---------- |
| 上电就卡死 / 复位       | 栈溢出、初始化错误、硬件引脚冲突 | 最高       |
| 操作某外设后异常        | 驱动 / 硬件时序 / 寄存器配置     | 高         |
| 多任务 / 中断后数据错乱 | 并发冲突、临界区未保护           | 高         |
| 长时间运行后崩溃        | 内存泄漏、资源耗尽               | 中         |
| 偶发卡死 / 数据错       | 时序竞争、内存越界               | 中高       |



### 4 分层验证

按照 软件 -> 驱动 -> 硬件的顺序排查。

1. 软件：屏蔽硬件操作，查看历史log，系统error信息等；
2. 驱动：单独验证驱动看是否异常；
3. 硬件：检查逻辑、时序是否正常，用万用表、示波器、逻辑分析仪检查信号是否正常。



### 5 DEBUG过程

DEBUG的时候首先要清楚业务模块是做什么的，功能是什么？是否独立？与哪些模块有交互？业务流程是什么？搞清楚这些东西，就可以继续往下了



### 6 平台差异

#### 裸机

* 核心痛点:栈溢出、中断冲突、寄存器配置错误、硬件时序不匹配、死循环。

- 调试器：J-Link/DAP-Link（必用）；
- 辅助工具：串口调试助手、示波器（测时序 / 电平）、万用表（测电源 / 引脚）；
- 软件手段：ASSERT 断言、栈溢出检测（在栈末尾加 “魔术字”，定期检查是否被改写）。

| 问题场景            | 具体排查方法                                                 |
| ------------------- | ------------------------------------------------------------ |
| HardFault 崩溃      | 1. 用调试器看 PC/LR 寄存器，定位崩溃的代码行；2. 检查栈指针 SP，看是否栈溢出；3. 排查空指针 / 野指针（比如未初始化就解引用）； |
| 中断不响应 / 乱触发 | 1. 检查中断使能寄存器、优先级配置；2. 看引脚是否有外部干扰（示波器测电平）；3. 排查中断服务函数是否有死循环； |
| 外设读写失败        | 1. 核对外设时钟使能（比如 I2C 没开时钟）；2. 用示波器测外设时序（如 UART 波特率是否匹配）；3. 检查引脚复用配置； |
| 程序跑飞            | 1. 看是否有数组越界（改写了程序计数器）；2. 检查看门狗是否误触发；3. 核对晶振 / 时钟配置（时钟错会导致代码执行速度异常）； |

#### FreeRTOS

* 核心痛点：任务优先级反转、栈溢出、信号量/队列死锁、中断中调用阻塞函数、共享资源竞争
* 调试工具：RTOS调试插件
* 辅助手段：RTOS自带的状态查询函数(vTaskList()/ vTaskGetRunTimesStats())、栈监控（uxTaskGetStackHighWaterMark()).

| 问题场景                         | 具体排查方法                                                 |
| -------------------------------- | ------------------------------------------------------------ |
| 任务卡死（一直 Running/Blocked） | 1. 用 `vTaskList()` 看所有任务状态，定位卡死的任务；2. 检查任务是否等待空队列 / 未释放的信号量；3. 核对任务优先级（是否低优先级抢占高优先级）； |
| 多任务数据错乱                   | 1. 检查共享变量是否加临界区保护（`taskENTER_CRITICAL()`/`xSemaphoreTake()`）；2. 排查中断是否改写了共享变量（未关中断）； |
| 中断中调用阻塞函数               | 1. 检查中断服务函数（ISR）是否调用 `vTaskDelay()`/`xSemaphoreGive()` 等阻塞函数；2. ISR 中只能用带 `FromISR` 后缀的函数（如 `xSemaphoreGiveFromISR()`）； |



#### Linux嵌入式

* 核心痛点：内存越界/泄漏、内核oops、驱动崩溃、多线程死锁、系统调用失败

* 调试工具：gdb/dmesg/strace

* 性能/内存工具：top/ps（进程状态）、free（内存使用）、KASAN（内存越界检查）、valgrind（内存泄漏）

* 内核调试：kdb/kgdb（内核调试）、printk（内核日志）。

| 问题场景        | 具体排查方法                                                 |
| --------------- | ------------------------------------------------------------ |
| 应用程序崩溃    | 1. 用 `gdbserver` 远程调试，看 core dump 文件（需开启 core 转储）；2. 编译时加 `-fsanitize=address` 开启 ASAN，检测内存越界；3. 用 `strace` 看系统调用是否失败（如 open 设备文件返回 -1）； |
| 内核 Oops/Panic | 1. 用 `dmesg` 看内核日志，解析 Oops 中的 PC 地址，定位崩溃的内核函数；2. 检查驱动代码（如空指针、寄存器越界访问）；3. 核对设备树配置（如引脚、中断号是否匹配）； |
| 内存泄漏        | 1. 用 `free` 看内存占用是否持续上涨；2. 用 `valgrind --leak-check=full ./app` 检测用户态内存泄漏；3. 内核态用 `kmalloc_debug` 追踪内存分配 / 释放； |
| 多线程死锁      | 1. 用 `pstack <pid>` 看线程调用栈，定位等待的锁；2. 用 `ps -T` 看线程状态（是否处于 D 状态）； |
